# Open-Source JavaScript Subtitle Libraries: Display, Editing & SRT Export — An Opinionated Evaluation

## TL;DR

- **No single open-source library does everything you want.** The winning architecture is a **combination**: build your own internal data model (a structured cue array, ideally Slate.js-shaped if you want rich-text editing) using a battle-tested parser/serializer like **subtitle.js** (MIT) for SRT/VTT round-tripping, and pick a display layer (a CSS overlay for simple styling, or JASSUB/ASS.js if you need broadcast-grade visual styling).
- **For the editing data model — your #1 criterion — the strongest foundation is Slate.js (MIT).** Its nested node tree with text "marks" (italic/bold as boolean properties on text leaves) is exactly the structured, inline-styling-aware representation you want. `slate-transcript-editor` shows the pattern but is dormant and is a _transcript_ editor (no SRT export), so treat it as a reference design, not a dependency.
- **SRT export is the easy part** (many MIT libraries do it); the hard parts you will almost certainly build yourself are **semantic/smart line-breaking, EBU/Netflix preset enforcement, and Whisper word-level import**. No JS library ships these as turnkey features.

## Key Findings

**The core tension in your requirements:** "display + editing + SRT + a robust internal model with inline italics" splits the ecosystem into three camps that don't overlap cleanly:

1. **Parsers/serializers** (subtitle.js, srt-parser-2, subsrt, vidstack/media-captions) — great data in/out, flat text model, no editing UI, no rich display.
2. **Renderers** (JASSUB/SubtitlesOctopus, ASS.js, imscJS, vtt.js) — excellent styled display, no editing, no SRT export.
3. **Rich-text editor frameworks** (Slate.js, and the transcript editors built on it) — excellent structured editable data model with inline styling, but not subtitle-aware and no SRT export out of the box.

Your "most important" criteria (data-model robustness + styled display) land in camps 3 and 2 respectively, while your mandatory SRT export lands in camp 1. That's why a combination is the realistic answer.

**On the internal data model (your single most important factor):** The key design question is how inline styling like italics is represented. There are three approaches in the wild:

- **Flat string with embedded tags** (e.g. `text: "<i>whisper</i>"`). This is what almost all SRT parsers do (subtitle.js, srt-parser-2, vidstack/media-captions). Simple, but italics are just characters in a string — you have to regex them, and it's fragile for editing.
- **Structured AST / rich-text model** (Slate.js): text is split into "leaves," each leaf carries boolean marks like `{ text: "whisper", italic: true }`. This is the gold standard for _editing_ because the editor, the toolbar commands ("toggle italic"), and serialization all operate on the same structured tree.
- **Format-specific structured model** (ass-compiler for ASS, imscJS's ISD for TTML): rich and accurate for that one format, but heavyweight and not designed as a general editing model.

For an editor with italics/bold/positioning/color commands, the Slate.js mark-based model is the best fit, and it's MIT-licensed and very actively maintained.

### Library-by-library findings

**Slate.js** (`slate`) — MIT. **31,679 GitHub stars, ~2,358,206 weekly npm downloads, 1,319 dependent projects**, actively maintained (latest **v0.124.1, published ~2 months before mid-2026**). **Editing data model only — not subtitle-aware.** Nested, recursive JSON node tree mirroring the DOM: element nodes (blocks/inlines) + text leaf nodes; inline formatting is implemented as **marks** (custom boolean/value props on text nodes, e.g. `bold: true`, `italic: true`), applied via `Editor.addMark`/`removeMark` and `editor.marks`. Schema-less core, plugin-first, designed so collaborative editing and custom serialization layer on top. **No SRT/VTT export, no display-over-video, no timing model** — you build the cue↔node mapping and the SRT serializer yourself. _Strength: the best structured editable model with first-class inline styling. Weakness: you build all subtitle-specific logic (timing, cues, export, display) on top._

**slate-transcript-editor** (`slate-transcript-editor`, pietrop; also Viostream and theirstory forks) — MIT. **77 stars**, 33 forks, **~64 npm downloads/week, last publish ~5 years ago (v0.1.6-alpha.19)**, Snyk rates maintenance **Inactive/possibly discontinued**. BBC-lineage (successor to `@bbc/react-transcript-editor`, which was Draft.js-based). Built on Slate.js. **Both display (synced media playback tied to text) and editing**, but it is a **transcript** editor, not a subtitle editor. Internal model is the "dpe" (digital paper edit) JSON: a flat `words[]` array (`{start, end, text}` in seconds) plus a `paragraphs[]` array with speaker/time, converted into the Slate model for editing. **Export formats are dpe JSON, Slate JSON, DOCX, TXT (with/without timecodes) — no SRT.** Notable real-world lesson documented by the author: maintaining word-level timecodes through edits is hard; they had to run realignment (`stt-align-node`) on save, which froze/crashed the UI on transcripts over ~1 hour, so they removed user-facing word-level timecodes and autosave. _Strength: a proven reference design for a Slate-based timed-text editor. Weakness: dormant, transcript-not-subtitle oriented, no SRT, performance caveats on long files._

**subtitle.js** (`subtitle`, gsantiago) — MIT. **29,363 dependents listed on npm, latest v4.2.2, last publish ~3 months before mid-2026**, actively maintained, TypeScript. **Parser/serializer only.** Clean node model: `{ type: 'cue', data: { start, end, text, settings } }` with timestamps in **milliseconds**; also `header` nodes. Stream-based (`parse`/`stringify`/`resync`/`map`/`filter`) plus sync helpers. **Exports SRT and WebVTT** (`stringify({ format: 'SRT' | 'WebVTT' })`). Inline italics are kept as raw tags inside the `text` string — **flat text model, no structured inline styling.** No editing UI, no display. _Strength: the cleanest, most reliable SRT/VTT round-trip library in JS; great as the I/O layer under your own model. Weakness: text is a flat string; no rich styling model._

**srt-parser-2** (`srt-parser-2`) — MIT. Parser/serializer, browser+Node. Array of `{ id, startTime, startSeconds, endTime, endSeconds, text }`. Tolerant of malformed SRT timestamps (pads/truncates digits, normalizes `.`→`,`). `fromSrt`/`toSrt`. SRT-focused (no VTT styling model). Flat text. _Good lightweight SRT-only fallback._

**subsrt** (`subsrt`, papnkukn) — parses/builds/converts **SUB, SRT, SBV, VTT, SSA/ASS, SAMI, LRC, JSON**. Flat caption model `{ start, end, text }`. Pluggable format handlers. Good as a multi-format converter; older and less actively maintained than subtitle.js. _Useful if you need many input formats funneled to SRT._

**subtitle-converter** (npm) — converts dfxp/scc/srt/ttml/vtt/ssa/ass; has a `removeTextFormatting` option and validation (overlap/timecode/formatted-text checks). Flat model. Handy converter, not an editor.

**vidstack/media-captions** (`media-captions`) — MIT. ~130 stars. Modern (~5kB) parser+renderer for **VTT, SRT, SSA/ASS**; TS types, ESM/CJS, works server-side, streams large files. Maps all formats to a VTT-extended cue model (`VTTCue` with `line`/`position`/`align`). Renders cues as a styled DOM overlay (CSS-customizable). **TTML/CEA-608/708 were listed by the maintainer as _planned_, not shipped** — verify current status before relying on it. **Display + parsing**, no editing UI. Powers Vidstack Player; Vidstack also ships a JASSUB plugin for libass-grade ASS. _Strength: clean modern display+parse with SRT in; good display layer. Weakness: VTT-centric model, no editing, ASS/TTML support partial/planned._

**JASSUB** (`jassub`) — the actively maintained libass-wasm fork. License is the libass stack: **LGPL-2.1-or-later AND (FTL OR GPL-2.0-or-later) AND MIT AND … ** (copyleft — important for your license review; it's usable but is _not_ permissive MIT). **~1,840 weekly downloads, latest v1.8.6 published ~2 months before mid-2026**, threaded, WASM, hardware-accelerated, WebGL/WebGPU. **Display only — renders ASS/SSA with near-full libass fidelity** (the most accurate styled subtitle rendering on the web). Use it when you need broadcast/anime-grade visual styling, karaoke, positioning, etc. _Strength: best-in-class styled display. Weakness: copyleft licensing, display-only, ASS-format-centric, heavy WASM payload._

**JavascriptSubtitlesOctopus / libass-wasm** (libass org) — same libass stack and license. The original; `libass-wasm` on npm is now effectively superseded (Snyk: no new npm versions in 12 months, "possibly discontinued"). **Prefer JASSUB.** Forks exist for Jellyfin, webOS, etc.

**ASS.js** (`assjs`, weizhenye) + **ass-compiler** (`ass-compiler`) — both MIT. ass-compiler (~2,178 npm dependents; last publish ~7 months before mid-2026) parses ASS into a structured JS object (`parse`/`compile`/`decompile`/`stringify`) — a genuinely usable structured model for ASS. ASS.js (latest **v0.1.4, last published ~a year before mid-2026**) renders ASS via the DOM; the project markets itself as much lighter than WASM solutions and relies on browser font-fallback so CJK "just works" — at "best-effort" rendering accuracy (not pixel-identical to libass). **Display (ASS.js) + structured parse (ass-compiler).** _Strength: MIT-licensed ASS handling with a real data model (ass-compiler) and lightweight DOM rendering (ASS.js); good CJK behavior. Weakness: ASS-centric; rendering not as exact as libass/JASSUB._

**imscJS** (`imsc`, sandflow/W3C-aligned) — **BSD-2-Clause** (permissive). ~510k npm downloads, last publish ~2 years before mid-2026 (mature/stable), the de-facto reference IMSC (TTML profile) renderer, MovieLabs-supported, documented on MDN. Parses IMSC/TTML 1.0.1 & 1.1 → ISD → HTML5. Handles EBU-TT-D, SMPTE-TT, etc. **Display + parse for TTML/IMSC**, no editing, no SRT export. _Strength: the authoritative TTML/IMSC renderer if you must show broadcast TTML. Weakness: TTML-only, render-focused, advanced ruby layout only fully works in Firefox._

**vtt.js** (Mozilla/videojs) — Apache/MIT lineage. WebVTT parser/processor used by Video.js for emulated text tracks. Display+parse for VTT; flat cue model; no SRT export, no editing. Largely a Video.js internal.

**Video.js / Plyr** — Apache-2.0 / MIT respectively. Both are **players** that display VTT captions via the native/emulated TextTrack + `VTTCue` model and CSS `::cue` styling (Plyr exposes caption color CSS vars; Video.js uses vtt.js). The WebVTT/`VTTCue` API itself gives you a structured-ish cue object (start/end/text + `line`/`position`/`align`) and supports `<i>/<b>/<u>/<c>/<v>` inline tags and `::cue` styling. **Display only; SRT is not natively supported by `<track>` (must convert to VTT first, e.g. via `srt-webvtt`).** Useful as the video shell; not your editing model.

**Whisper import (wishlist):** No JS library ingests Whisper word-level output as a first-class feature. In Node, `nodejs-whisper` can emit SRT/VTT/JSON directly (set `wordTimestamps: true`), and WhisperX/faster-whisper produce word-level JSON. The community pattern is roll-your-own: parse Whisper JSON `segments[].words[]` (`{start, end, word}` in seconds) into your cue model. The notable real-world gotcha (documented in the OpenAI Whisper repo) is that Whisper's `words` and `text` can mismatch around pauses, causing flashing/odd cues. **You will write this importer yourself**, but it's straightforward against a structured cue model.

**Smart/semantic line-breaking (wishlist):** This is an active research area, not a solved JS library. Academic work uses Chink-Chunk heuristics, CRF/SVM classifiers, and LSTM/seq2seq models (e.g. Álvarez et al.; Karakanta et al. "MuST-Cinema"). The consensus rule ("don't split an article from its noun; break at syntactic boundaries") is well documented but **no maintained npm package implements it**. Python pipelines (e.g. whisperx-subtitles-replicate) use PySBD for sentence segmentation. **You'll build this** — either rule-based (punctuation + a small POS/conjunction heuristic) or by calling an NLP service.

**Broadcast/streaming presets (wishlist):** No JS library enforces EBU/Netflix/BBC presets. The rules are public and codifiable. Per Netflix's **English (USA) Timed Text Style Guide**, reading speed can be up to **20 characters per second for adult content and 17 CPS for children's content** (note: the 17-CPS figure is the children's limit for English; some other languages use 17 CPS as the adult limit — check the language-specific guide). Per Netflix's **General Requirements**: **42 characters per line for most Latin-alphabet languages**, minimum duration **5/6 (five-sixths) of a second** (e.g. 20 frames at 24fps), maximum **7 seconds**, and a **minimum 2-frame gap between consecutive subtitle events regardless of frame rate**; delivery in TTML (IMSC1.1 for Japanese). EBU-TT-D / EBU-STL are EBU's formats. For conversion to/from EBU-STL and TTML, the best open tool is **sandflow/ttconv** (BSD, Python, not JS) which round-trips STL/SRT/TTML/SCC/WebVTT through a TTML-2-shaped canonical model with filters. **In JS you'll implement preset validation (CPS/CPL/duration/gap checks) yourself** against your cue model — these are simple arithmetic rules once your model has ms-accurate timing and character counts.

## Recommendation (opinionated)

**Build a small custom internal data model and assemble three open-source pieces around it. Do not try to adopt one all-in-one library — none exists that meets your bar.**

**Stage 1 — Define your own cue model (this is your core asset).** Make it Slate-shaped so editing and inline styling are first-class. Concretely: an array of cues, each `{ id, start, end (ms), region/position, children: [ Slate text leaves ] }` where a leaf is `{ text, italic?, bold?, color? }`. This directly satisfies criterion #1 (robust model with control commands for italics and other inline styling) and is the same proven pattern slate-transcript-editor and flycut-caption use. ms-accurate timing makes preset validation trivial later.

**Stage 2 — Editing UI: Slate.js (MIT).** Use Slate's marks for italic/bold/color toggles and your own block element per cue. This is the single best-supported, actively maintained structured editor in JS (31,679★, ~2.36M weekly downloads). Use `slate-transcript-editor` only as a _reference implementation_ for cue↔node mapping and the timecode-preservation problem — don't depend on it (dormant, 77★, ~64 downloads/week, no SRT, perf issues >1 hour). If you'd rather start from something subtitle-shaped and MIT-licensed, **flycut-caption** (structured `SubtitleChunk[]`, SRT export, React 19, video preview + burn-in, ASR-oriented, ~1.3k★) and **laubonghaudoi/subtitle-editor** (Next.js 15/React 19, SRT/VTT, waveform, video preview, ~35★) are the two most credible 2025-era editors to fork — but verify flycut-caption's "MIT + mandatory brand/attribution" clause against your needs, as it restricts removing its logo/watermark.

**Stage 3 — SRT/VTT I/O: subtitle.js (MIT).** Use it as the serialization boundary: convert your model → subtitle.js cue nodes → `stringify({ format: 'SRT' })`. It's the cleanest, most reliable round-trip library and also gives you VTT for free. Write a thin adapter that flattens your Slate leaves to `<i>…</i>` on SRT export and re-parses them on import.

**Stage 4 — Display layer: pick by styling ambition.**

- **Simple/standard styling presets** (font, color, position, italics): render your cues as a **CSS DOM overlay** yourself (what vidstack/media-captions does in ~5kB) or just use vidstack/media-captions directly. Cheapest, MIT, full styling control, easy preset theming.
- **Broadcast/anime-grade visual fidelity** (exact positioning, karaoke, complex ASS): add **JASSUB** for display — but note its **copyleft (LGPL/GPL) licensing**, which matters for your license review and may be a reason to avoid it if you need fully permissive deps. A permissive alternative is **ASS.js (MIT)** at slightly lower rendering fidelity.
- **Must consume/show broadcast TTML/IMSC or EBU-TT-D**: add **imscJS (BSD-2)** for that specific display path.

**What you will build yourself (budget for these — no library provides them):**

1. **Whisper word-level import** — parse Whisper/WhisperX JSON `segments[].words[]` into your cue model; handle the words-vs-text mismatch/flashing gotcha. Straightforward.
2. **Semantic/smart line-breaking** — start rule-based (CPL limit + don't break article-noun/verb-object, prefer breaks at punctuation/conjunctions); upgrade to an NLP/ML approach later if needed. This is research-grade and the main custom-engineering risk.
3. **EBU/Netflix/BBC preset enforcement** — encode the public numeric rules (Netflix: 20 CPS adult / 17 CPS children's for English, 42 CPL, 5/6s–7s duration, 2-frame gap) as validators over your ms-accurate model. Simple arithmetic once the model exists.
4. **EBU-STL / TTML export** if you need broadcast delivery — there is no good JS writer; either port logic or run **sandflow/ttconv** (BSD, Python) as a server-side microservice (fits your PHP/LAMP+Node backend; keeps data on your own Swiss/EU infrastructure).

**Why this combination and not alternatives:** Adopting a renderer-first stack (JASSUB/ASS.js) would give you beautiful display but force ASS as your data model — awkward for SRT-first editing and copyleft in JASSUB's case. Adopting a parser-first stack (subtitle.js alone) gives you SRT but a flat text model that fights you the moment you add an italics toolbar. Slate gives you the editing model you ranked #1; subtitle.js gives you the mandatory SRT cleanly; a CSS overlay or vidstack gives you styled display with preset theming. Each piece is independently maintained and (except optional JASSUB) permissively licensed.

## Caveats

- **Licensing:** Slate, subtitle.js, ass-compiler/ASS.js, vidstack/media-captions, srt-parser-2, Plyr are MIT; imscJS is BSD-2; Video.js is Apache-2.0 — all permissive. **JASSUB/SubtitlesOctopus/libass-wasm are LGPL-2.1+/GPL-2.0+ (copyleft)** — usable but review obligations before bundling. **flycut-caption is "MIT plus a brand/attribution clause"** that forbids removing its UI logo/watermark — not pure MIT; confirm acceptability.
- **Maintenance reality:** slate-transcript-editor is effectively abandoned (last npm ~5 years ago, ~64 downloads/week); libass-wasm (original) is stale — use JASSUB instead; flycut-caption and laubonghaudoi/subtitle-editor are new (2025) and relatively small (so smaller community/long-term-support risk). Star counts and publish dates cited are as of mid-2026 and move; re-check before committing.
- **"Both display and editing" in one OSS package** essentially only exists today in the new small editors (flycut-caption, laubonghaudoi/subtitle-editor) and the dormant slate-transcript-editor — hence the build-your-own-model recommendation.
- **Data residency:** every recommended piece runs client-side or in your own Node/PHP backend; the only server-side suggestion (ttconv) is self-hostable, so a Swiss/EU-only deployment is achievable with no third-party SaaS dependency.
- **Whisper word-vs-text mismatch** is a known source of flashing/duplicated cues — validate imported cues.
- **Netflix CPS nuance:** the 20-CPS-adult / 17-CPS-children's split is the English (USA) guide; other languages have their own language-specific limits (some use 17 CPS as the adult ceiling). Always validate against the specific language guide if you implement Netflix presets.
- I did not find any maintained JS library that natively enforces EBU R128 / EBU-STL writing or Netflix TTML validation; those remain custom or Python-side (ttconv) work.
