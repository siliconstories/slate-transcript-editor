# Open-Source Repos for a Waveform-Only Video Timeline Scrubber

## TL;DR

- **wavesurfer.js (BSD-3-Clause, JS)** is the single best pick for most developers: one library that both renders the interactive waveform scrubbing track (playhead, click-to-seek, drag-to-scrub, hover preview, zoom, region selection via plugins) **and** can attach to your `<video>` element so the cursor stays synced to video playback. Pair it with **bbc/audiowaveform** to pre-compute peaks for long videos.
- If you need a purpose-built long-form editor UI (separate overview + zoom views, segment/point markers), **Peaks.js (LGPL-3.0, JS, by BBC R&D)** + **bbc/audiowaveform (GPL-3.0, C++ CLI)** is the strongest "generation + display" combo — it was designed for scrubbing/editing hour-long media with server-pre-generated waveform data.
- **PHP cannot realistically generate a waveform from video in pure PHP.** Every viable server-side route shells out to ffmpeg (to extract/decode the audio track) and/or the BBC audiowaveform binary. Use **PHP-FFMpeg (MIT)** for a static PNG or **podcasthosting/audiowaveform-php-client (MIT)** to produce peak JSON/.dat for a JS scrubber; avoid the abandoned, unlicensed pure-PHP packages.

## Key Findings

The job splits into two layers, and the core architectural decision is **one library that does both** (wavesurfer.js) vs. **a generation + display pair** (Peaks.js + audiowaveform). For video specifically, two constraints recur: (1) the visual track is the _audio waveform extracted from the video_, so you always need the audio track, and (2) decoding a full-length video in the browser is expensive and memory-hungry, so **pre-computed peak data is strongly preferred for anything beyond a few minutes**.

### Category 1 — Timeline / cursor representation & interactivity (the UI layer)

**Top recommendation: wavesurfer.js.** It renders the waveform on a canvas and layers interaction on top. Click-to-seek and drag-to-scrub are built in; official plugins add the rest:

- **Timeline** — notches and time labels below the waveform.
- **Regions** — draggable/resizable overlays for region selection and looping.
- **Hover** — a vertical line and timestamp on hover (replaces the old Cursor plugin).
- **Minimap** — a small overview waveform that acts as a scrollbar.
- **Zoom** — scroll-to-zoom.

For this exact use case, you pass an existing `<video>` element via the `media` option, and wavesurfer drives/stays synced to the video's playback time instead of playing its own audio. Repo: github.com/katspaugh/wavesurfer.js. License: **BSD-3-Clause** (permissive). Language: vanilla JS/TypeScript; official wrappers exist for React (`@wavesurfer/react`, BSD-3-Clause) and Vue. Maintenance: **very active — v7.12.7, released May 13, 2026, with 10.2k GitHub stars.** Caveat: v7 renders into a Shadow DOM (style via `::part()`), and large/long files should use pre-decoded peaks (see Category 2).

**Alternative for long-form editing UIs: Peaks.js.** BBC R&D's component is purpose-built for interacting with long media: a zoomable detail view plus an overview pane, point and segment markers, click-to-seek, dragging, and zoom/scroll, all synced to a supplied `<audio>` or `<video>` `mediaElement`. Repo: github.com/bbc/peaks.js (active development continues there by Chris Needham, BBC R&D; a Codeberg mirror exists). License: **LGPL-3.0** (per its COPYING file). Language: vanilla JS; depends on Konva.js and waveform-data.js. Maintenance: active — **latest tag is v4.0.0-beta.2, a pre-release dated late April / May 2026; 3.4k GitHub stars** (the stable 3.x line installs by default from npm). Caveat: copyleft license; heavier dependency footprint than wavesurfer; pre-release for the v4 line.

**React transcript ecosystem (context, not a primary pick):** **pietrop/slate-transcript-editor** and the older **bbc/react-transcript-editor** are timed-text editors that sync a transcript to media playback (word-level timecodes), aimed at correcting STT output. They are part of the same "edit long media in the browser" lineage but are about transcript scrubbing, **not** waveform scrubbing — relevant only if your scrubber also needs a synced transcript. The DOTE project (bigsoftvideo) is a related transcription environment that does combine a draggable waveform timeline with video, if you want to study a full reference implementation.

### Category 2 — Waveform generation & display

**Client-side, real-time (short clips):** Both wavesurfer.js and Peaks.js can decode audio in the browser via the Web Audio API and draw the waveform live. Per both projects' own FAQs, this is fine for short audio but CPU-intensive and can fail to decode large files due to memory limits — and for video you'd download/decode the entire file. Use this only for short clips.

**Pre-computed peaks (recommended for video):**

- **bbc/audiowaveform** — a **C++ command-line application**; per its README it "generates waveform data from either MP3, WAV, FLAC, Ogg Vorbis, or Opus format audio files," outputting compact peak data as binary `.dat` or `.json`, and can also render PNG images at a given zoom/offset. It pairs natively with both Peaks.js and wavesurfer.js (both accept pre-computed peaks). License: **GPL-3.0; latest release 1.10.3 (Aug 20, 2025); 2,123 GitHub stars.** For video you pipe through ffmpeg: `ffmpeg -i video.mp4 -f wav - | audiowaveform --input-format wav --output-format json -b 8 > peaks.json`. Caveat: it does not read video itself (ffmpeg must decode to a WAV stream first), and piping in is supported only for MP3/WAV/raw audio (not FLAC/Ogg).
- **bbc/waveform-data.js** — the JS companion (LGPL-3.0) that consumes those `.dat`/`.json` peak files or generates peaks via Web Audio. It is what Peaks.js uses under the hood; use it directly if you want to draw a custom canvas/SVG track yourself.

**Node helper:** **t4nz/ffmpeg-peaks** generates peak JSON from an audio file via ffmpeg if you'd rather keep peak generation inside a Node service.

### Category 2b — PHP server-side options (honest assessment)

There is **no robust pure-PHP waveform generator for video**. Pure-PHP options only read WAV/MP3 (you must extract audio first), and the maintained ones all shell out to native binaries:

- **PHP-FFMpeg/PHP-FFMpeg** — actively maintained OO wrapper around the ffmpeg/ffprobe binaries (MIT, ~5k stars, v1.4.0 released Jan 19, 2026, PHP 8.0+). Per its docs: _"You can generate a waveform of an audio file using the FFMpeg\Media\Audio::waveform method… The output file MUST use the PNG extension… If you want to get a waveform from a video, convert it in an audio file first."_ So it outputs a **PNG image only, from audio only** — good for a static thumbnail track, not for interactive peak data.
- **podcasthosting/audiowaveform-php-client** — the best PHP route for **peak JSON/.dat** feeding a JS scrubber. **MIT; v0.2.2 (Nov 13, 2025), by Fabio Bacigalupo; requires symfony/process ^7.3.** Per Packagist it "implements all the arguments from the commandline audiowaveform command as methods. The script calls the shell command" — i.e., it drives the bbc/audiowaveform binary (which must be installed). Audio in; video via the ffmpeg pipe above. Caveat: low adoption / pre-1.0.
- **maximal/audio-waveform-php** — pure-ish PHP using **SoX + the GD extension**; has a `getWaveformData()` method whose README explicitly says it can "integrate with different libraries, for instance, with the great wavesurfer.js." **MIT, but stale (v1.2.1, last released Feb 14, 2019; ~14 stars, single contributor); audio-only.** Usable for simple cases but unmaintained.
- **Avoid for production:** **boyhagemann/waveform** (pure-PHP, WAV-only; _"no license information available"_ per Packagist; last updated Nov 23, 2016 — abandoned and legally unsafe) and **alexkirs/php-waveform-json** (JSON peaks from MP3 via the LAME encoder; no license, ~2 commits, abandoned, not Composer-installable).

## How the two layers fit together

- **One-library approach (simplest):** wavesurfer.js _is_ both the generator and the display. Point it at your `<video>`; for short clips it decodes the audio and draws the track itself. The cursor, click-to-seek, drag-scrub, hover and regions all come from wavesurfer + its plugins, synced to video playback.
- **Generation + display pair (scales to long video):** pre-generate peaks once on the server with **bbc/audiowaveform** (fed by ffmpeg from the video), store the small `.json`/`.dat` alongside the video, and load it into either **wavesurfer.js** (`peaks` option) or **Peaks.js**. This is the standard pattern for hour-long media: the waveform paints instantly (~tens of KB instead of multi-MB decode) and the UI library handles all interaction.

## Recommendations

1. **Start here:** wavesurfer.js v7 + the **Timeline**, **Regions**, and **Hover** plugins, attached to your `<video>` element via the `media` option. For clips under ~3–5 minutes, let it decode in-browser. This satisfies the entire feature set (waveform-only track, playhead, click-to-seek, drag, hover preview, zoom, regions, video sync) in one BSD-licensed dependency.
2. **Threshold that changes the plan — switch to pre-computed peaks** the moment videos exceed a few minutes, or as soon as you observe slow first paint, inaccurate waveforms, or browser decode failures. Generate peaks server-side: `ffmpeg -i video.mp4 -f wav - | audiowaveform --input-format wav --output-format json -b 8 > peaks.json`, then pass the JSON to wavesurfer's `peaks` option. Use `-b 8` (8-bit) for small files; Peaks.js specifically requires 8-bit.
3. **Choose Peaks.js instead of wavesurfer** only if you need its richer editor UX (separate overview + zoom panes, segment/point annotation). Budget for the LGPL-3.0 (Peaks.js) / GPL-3.0 (audiowaveform) licensing if you redistribute.
4. **PHP backend:** to feed a JS scrubber, produce peak JSON via **podcasthosting/audiowaveform-php-client** (or call the audiowaveform binary directly with `exec`/`shell_exec` and `symfony/process`); use **PHP-FFMpeg** only if a static PNG track is acceptable. Do **not** depend on boyhagemann/waveform or alexkirs/php-waveform-json (no license, abandoned).
5. **Licensing decision rule:** if you ship a closed-source product and want zero copyleft obligations, stay on the wavesurfer.js + (server-side) audiowaveform-via-CLI path — wavesurfer is BSD-3-Clause, and merely _invoking_ the GPL audiowaveform binary as a separate process (rather than linking its code) keeps your application code unaffected. If you embed Peaks.js or waveform-data.js (LGPL-3.0) in the browser bundle, comply with LGPL relinking terms.

## Caveats

- **"Video support" means audio-track sync, not video frames.** wavesurfer.js and Peaks.js are audio-visualization libraries; with a `<video>` element they draw that video's _audio_ waveform and sync the playhead to `currentTime` — they do not render video thumbnails on the timeline. If you also want a filmstrip of video frames, that's a separate feature (e.g., ffmpeg thumbnail extraction).
- **No tool reads a waveform from video directly.** Audio must be extracted/decoded first — client-side via Web Audio, or server-side via ffmpeg before audiowaveform/SoX.
- **Pure-PHP waveform generation is essentially a dead end for video** and largely unmaintained even for audio; treat the maintained PHP options as thin wrappers over ffmpeg/SoX/audiowaveform.
- **Version/stars are point-in-time** (verified mid-2026): wavesurfer.js v7.12.7 / 10.2k★; Peaks.js v4.0.0-beta.2 pre-release / 3.4k★; audiowaveform 1.10.3 / 2,123★. The Peaks.js v4 line is still pre-release — pin to a stable 3.x release if you want maximum stability today.
- **VBR audio mismatch:** wavesurfer notes that variable-bit-rate files can desync the waveform from playback; convert to CBR or use its Web Audio shim if you see drift.
